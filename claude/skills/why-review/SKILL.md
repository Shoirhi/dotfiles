# Why-Focused PR Review

AI時代のコードレビュースキル。Howではなく**Why**を問う。
専門エージェントが多角的に分析し、レビュアーに**視点を提示**する。最終判断はレビュアーが行う。

## Usage

```
/why-review <PR番号> [--focus=comments|tests|errors|types|code|simplify]
```

- PR番号が未指定の場合、現在のブランチのPRをレビューする
- `--focus`が未指定の場合、全エージェントによるフルレビューを実行する

## 背景

AIがタイポ、構文エラー、パフォーマンス問題を検出できる時代において、人間のレビューの価値は「意図を問うこと」にある。AIはビジネス文脈や将来計画を理解していない。このスキルは、専門エージェントによる多角的な分析結果を**視点**としてレビュアーに提示し、Why視点でのレビューを支援する。

## Steps

### 1. PR情報の収集（並列実行）

以下のコマンドを並列で実行し、PRの全体像を把握する:

- `gh pr view <PR番号> --json title,body,author,baseRefName,headRefName,labels,files,additions,deletions,commits` -- PRのメタデータ
- `gh pr diff <PR番号>` -- 実際の差分
- `gh pr view <PR番号> --json comments,reviews` -- 既存のコメントとレビュー

PR番号が未指定の場合は `gh pr view --json number` で現在のブランチのPR番号を取得する。

### 2. プロジェクトガイダンスの収集

レビューの基準となるプロジェクト規約を確認する:

- `CLAUDE.md` -- プロジェクト固有の指針
- lint設定（`.eslintrc`, `.stylelintrc`等）
- TypeScript設定（`tsconfig.json`）
- その他の規約ファイル（`.editorconfig`, `.prettierrc`等）

### 3. コードの理解

レビューの前提として、コードが何をしているかを把握する。差分を読み、以下を整理する:

- 変更されたファイルと、その役割
- 変更の概要（何が追加・削除・修正されたか）
- 影響範囲（他のモジュールへの波及）

この理解を「変更サマリー」としてレビューの冒頭に簡潔にまとめる。

### 4. 専門エージェントによる多角的分析（並列実行）

`--focus`フラグに応じて、該当するエージェントを並列で起動する。フラグ未指定時は全エージェントを起動する。

| focus値 | エージェント | 分析観点 |
|---------|------------|---------|
| `code` | `code-reviewer` | コード品質、設計パターン、保守性 |
| `comments` | `comment-analyzer` | コメントの正確性、完全性、コメント腐敗リスク |
| `errors` | `silent-failure-hunter` | サイレント失敗、握りつぶされたエラー、不適切なフォールバック |
| `types` | `type-design-analyzer` | 型設計、カプセル化、不変条件の表現 |
| `simplify` | `code-simplifier` | 簡素化の余地、冗長性、可読性の改善 |

各エージェントには以下を渡す:
- PRの差分全体
- 変更されたファイルの一覧
- 変更サマリー（Step 3で整理した内容）
- プロジェクトガイダンス（Step 2で収集した規約）

重要: 全エージェントは**分析と視点の提示のみ**を行う。ファイルの編集・書き込みは一切行わない。エージェント起動時にこの制約を明示的に指示すること。

### 5. Why視点での分析

エージェントの分析結果を踏まえ、以下のWhy観点で差分を精査する。全てに言及する必要はない。該当する観点だけをピックアップする。

#### 5a. 意図の確認（Why）

- なぜこの実装方法を選んだのか？他のアプローチと比較検討したか？
- なぜこのライブラリ・パターン・アーキテクチャを採用したのか？
- PR descriptionやコミットメッセージから意図が読み取れるか？

#### 5b. ビジネス文脈・将来設計

- この変更はビジネス要件を正しく反映しているか？
- 将来の拡張性を考慮した設計になっているか？過度な最適化になっていないか？
- ビジネスドメインにおける暗黙のルール（命名規則、業界慣行、法規制）に適しているか？

#### 5c. セキュリティ

- 認証・認可の処理は適切か？
- ユーザー入力のバリデーションは十分か？
- 機密データの取り扱いに問題はないか？

#### 5d. エッジケース

- 並行処理・競合状態
- 境界値（空配列、null、巨大データ）
- エラー時のフォールバック
- 外部サービス障害時の挙動

### 6. 結果の集約

全エージェントの分析結果とWhy分析を統合する:

1. **重複排除**: 複数エージェントが同じ箇所を指摘している場合、統合して1つにまとめる
2. **信頼度フィルタ**: 信頼度80%未満の指摘は除外する
3. **severity分類**: 各指摘を以下の3段階に分類する

| severity | 基準 | 例 |
|----------|------|-----|
| **Critical** | バグ、セキュリティ脆弱性、データ損失リスク | 未検証の入力値、認証バイパス、レースコンディション |
| **Important** | テスト不足、品質上の問題、規約違反 | カバレッジの欠落、エラーハンドリング不足、型安全性の欠如 |
| **Advisory** | 改善提案。`--focus`で明示的に要求された場合のみ報告 | リファクタリング候補、命名改善、簡素化の余地 |

### 7. レビュー視点の出力

以下の構造でレビュアーへの視点提示をまとめる:

```markdown
## PR Review: #<PR番号> <PRタイトル>

### 変更サマリー
（Step 3で整理した内容を簡潔に）

### 検討していただきたい視点

#### Critical
（severity=Criticalの指摘。該当なしの場合は「該当なし」と明記）

#### Important
（severity=Importantの指摘）

#### Advisory
（--focusで明示要求された場合のみ。該当なしの場合はセクション自体を省略）

各視点は以下のフォーマットで記載:

**[label]** ファイル名:行番号
- 観察: （何が起きているか、事実の記述）
- 視点: （なぜ気になるか、検討してほしい理由）
- 検討案: （もし改善するならどうするか。あくまで一案として提示）

### 総評
- 全体的な印象
- 特に良かった点（[praise]を忘れずに）
- レビュアーとして特に注目してほしい箇所
```

## Labels

各視点に以下のラベルを付与する:

| ラベル | 意味 |
|---|---|
| `[question]` | 意図を教えてほしい質問。Whyが読み取れない箇所 |
| `[concern]` | 懸念事項。確認・検討を推奨 |
| `[suggestion]` | 検討してほしい提案。採否はレビュアー判断 |
| `[praise]` | 良い点の共有。チームの学びになる設計や実装 |
| `[nit]` | 些細な観察。対応は完全に任意 |

## コミュニケーションの原則

目的は「レビュアーに有用な視点を提供すること」。

- 指示ではなく視点を提示する。「すべきです」ではなく「この点についてどうお考えですか」
- 主語は常に「コード」であり「人」ではない
- NG: 「あなたはバリデーションを追加すべきです」
- NG: 「バリデーションが必要です」（断定的な指示）
- OK: 「この入力値のバリデーションについて、意図を確認したいです。現状では外部入力がそのまま渡されているように見えます」

## Rules

- Howの指摘（フォーマット、命名規則、パフォーマンスの微細な改善）は、AIツールに任せる。ただし、設計意図に関わるものは例外
- 全ての観点に無理にコメントしない。該当しないものはスキップする
- 信頼度80%未満の指摘は報告しない。確信が持てない場合は`[question]`として意図を確認する
- Advisory（改善提案レベル）は`--focus`で明示的に要求された分析領域についてのみ報告する。`--focus`未指定のフルレビューではCriticalとImportantのみ出力する
- `[praise]`は忘れずに。良いコードを見つけたら積極的に共有する。チームの学びになる
- レビュー結果はターミナルに出力する。`gh pr review`コマンドでGitHubに投稿はしない（ユーザーが明示的に依頼した場合のみ投稿する）
- エージェントはファイルの読み取りと分析のみを行う。コードの編集・書き込みは一切行わない
- 最終判断は常にレビュアーが行う。スキルの役割は視点の提示であり、判定ではない
